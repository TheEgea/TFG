\section{NIST SP 800-30 Five-Phase Risk Assessment}

This section details the systematic risk assessment following NIST SP 800-30\cite{nist_sp80030}.
The summary risk matrix is provided in Table~\ref{tab:risk_matrix} in Chapter~\ref{ch:feasibility}.

\subsection{Phase 1: Asset Identification and Scope Definition}

Critical assets are categorised by CIA triad:

\textbf{Hardware Assets}:
\begin{itemize}
\item HomeLab server (2$\times$ Xeon E5-2680v4, 64~GB RAM) --- Availability critical
\item Development workstations (laptop + desktop) --- Integrity critical
\item Network equipment (router, switch) --- Availability critical
\item External storage (backup) --- Confidentiality + Availability critical
\end{itemize}

\textbf{Software Assets}:
\begin{itemize}
\item EVE-NG Community Edition --- Availability critical
\item Parrot OS Security distribution --- Integrity critical
\item Vulnerable VMs (Metasploitable, DVWA) --- Integrity critical
\item Development tools (VSCode, Git, LaTeX) --- Integrity critical
\end{itemize}

\textbf{Data Assets}:
\begin{itemize}
\item Thesis source code and documentation --- Confidentiality + Integrity critical
\item Lab topology configurations --- Integrity + Availability critical
\item Student learning materials --- Availability critical
\item Project research findings --- Confidentiality + Integrity critical
\end{itemize}

\textbf{Scope}: Infrastructure and technical risks within the controlled laboratory
environment. Excluded: organisational, regulatory, and external threats beyond the
academic context.

\subsection{Phase 2: Threat and Vulnerability Identification}

\textbf{Threat categories (NIST SP 800-30)}:
\begin{itemize}
\item \textbf{Hardware Threats}: Disk failures, power supply degradation, CPU overheating,
      memory exhaustion
\item \textbf{Software/Configuration Threats}: Outdated VM images, incompatible versions,
      configuration drift, EVE-NG crashes
\item \textbf{Knowledge Gap Threats}: Insufficient expertise in advanced networking,
      virtualisation optimisation, security best practices
\item \textbf{Resource Constraint Threats}: Limited RAM for simultaneous VM operations,
      storage capacity limits
\item \textbf{Integration Threats}: EVE-NG network compatibility issues, tool interoperability
\item \textbf{Data Loss Threats}: Accidental deletion, corruption, hardware failure
\end{itemize}

\textbf{Key vulnerabilities}:
\begin{itemize}
\item Aging server components (end-of-life approaching)
\item Single point of failure --- no infrastructure redundancy
\item EVE-NG Community Edition has no commercial support
\item Limited RAM capacity for production-scale virtualisation
\item Inadequate documentation of network topology state
\item Limited backup redundancy (single backup location)
\end{itemize}

\subsection{Phase 4: Risk Evaluation and Prioritisation}

\textbf{HIGH Priority}: EVE-NG Performance Issues --- infrastructure bottlenecks causing
lab functionality failures. Medium probability, high impact. Requires immediate resource
optimisation and monitoring implementation.

\textbf{MEDIUM Priority}:
\begin{enumerate}
\item VM Incompatibility --- disparate image formats causing integration problems
\item Knowledge Gaps --- experience gaps in advanced networking/virtualisation
\item Configuration Documentation --- inadequate topology documentation
\item Data Loss --- insufficient backup redundancy (single point of failure)
\end{enumerate}

\textbf{LOW Priority}: Hardware Degradation --- aging components manageable through
preventive maintenance.

\subsection{Phase 5: Mitigation Strategies}

\textbf{EVE-NG Performance Issues (REDUCTION)}:
\begin{itemize}
\item Hardware dimensioning: 64~GB RAM dedicated to the server running EVE-NG
      (32~GB proved insufficient --- additional module acquired, \textasciitilde{}80\,\euro)
\item Resource optimisation: memory tuning, VM suspension when idle
\item Documented fallback: VirtualBox as alternative hypervisor if critical failure
\item \textit{Result}: Probability reduced from Medium to Low
\end{itemize}

\textbf{VM Incompatibility (PREVENTION)}:
\begin{itemize}
\item Early testing: validate all images in target environment before use
\item Format standardisation: QCOW2 for EVE-NG compatibility
\item Alternative repositories maintained (VulnHub, GitHub)
\item \textit{Result}: Probability reduced from Medium to Low
\end{itemize}

\textbf{Knowledge Gaps (PREVENTION + REDUCTION)}:
\begin{itemize}
\item Structured learning: Task 1.2 (15h) comprehensive framework research
\item Continuous upskilling during lab research phases (Tasks 2.1--3.1)
\item Expert consultation with tutor Pere Vidiella i Catalan
\item Technical references: NIST\cite{nist_sp800115}, OWASP\cite{owasp_testing_guide}
\item \textit{Result}: Impact reduced from Medium to Low-Medium
\end{itemize}

\textbf{Configuration Documentation (PREVENTION)}:
\begin{itemize}
\item Continuous documentation: update topology diagrams during development
\item Automated export: EVE-NG built-in topology export
\item Version control: all network configs via Git repository
\item \textit{Result}: Probability reduced from Medium to Low
\end{itemize}

\textbf{Data Loss (REDUCTION)}:
\begin{itemize}
\item Daily automated encrypted backups to external storage
\item Version control: full Git history on GitHub
\item Off-site cloud backup (student account)
\item \textit{Result}: Probability Low, Impact Low
\end{itemize}

\textbf{Hardware Degradation (PREVENTION)}:
\begin{itemize}
\item Quarterly hardware diagnostics (disk health, temperature monitoring)
\item Automated S.M.A.R.T. alerts for disk errors and CPU temperature thresholds
\item Component procurement plan with 6-month horizon for critical parts
\end{itemize}

\textbf{Risk Assessment Conclusion}: All identified risks have been systematically
evaluated and assigned concrete mitigation strategies. No individual risk is
considered project-blocking.
